## Ярус 2 — контракт входа ⚠ ТРЕБУЕТ РЕШЕНИЯ (Р1)

Сейчас: `np.maximum(arr, 0.0)` молча уничтожает знак прокси. Вся нормализация
(`ref_scale_ema`, rel_vol, shock) предполагает положительную шкалу — «просто убрать клип» нельзя.

- **Вариант A (рекомендуемый для починки):** контракт `proxy ≥ 0`, отрицательный вход → `ValueError`
  (fail fast вместо тихой потери сигнала). README: как готовить прокси (softplus, не raw hinge).
- **Вариант B (исследовательский):** shift-инвариантная внутренняя нормализация для знакопеременных
  прокси. Тянет за собой все шкалы → НЕ в починку, оформить как research note в backlog A1.

**Приёмка:** тест — отрицательный прокси падает с понятной ошибкой; положительные пути не изменились.

## Дополнение 2026-06-09b: калибровка без устаканивания + survival-метрика (→ ярус 5)

Контекст: миссия проекта переформулирована владельцем — «держать систему на
седловине как можно дольше, не ухудшая динамику; качество не должно страдать
(улучшение — побочный бонус)». Несходимость калибровки ≠ невозможность
юстировки: это несоответствие горизонта измерения структуре сигнала.

### Цепочка фоллбэков калибровки (НЕ или/или)

| Стадия | Механизм | Когда |
|--------|----------|-------|
| 0 | Штатный settle-check (CV+growth) → снап | сигнал устаканился |
| C1 | Период-aware окно: автокорреляция shock-ряда в фазе B → оценка доминирующего периода → окно ≥ периода → повторный settle-check | периодическая структура (кейс multiplayer_4: период ~400 vs окно 40) |
| C2+C3 | Робастный квантильный снап по z-сэмплам ВСЕГО warmup + градуированный запас: deadband = q90 × (1 + κ·instability). Settle-check и взятие снапа разъединяются: снап возможен и без устаканивания. Steering остаётся живым | статистика не устаканилась, но квантили осмысленны |
| last resort | Текущий пин deadband = max (v10.9 degraded) | квантили — мусор: монотонный рост (ловится growth-критерием), истинная нестационарность всех масштабов |

Попутно C0: переименовать `_warmup_converged` → `_calibration_settled`
(терминологическая коллизия «convergence» провоцирует неверное прочтение).

### C4: онлайн re-base deadband — ОТЛОЖЕН, гейтится A1

Анализ рисков (2026-06-09): re-base опаснее, чем выглядит.
1. **Boiling frog:** база, плывущая за q90(|z|), поглощает медленно растущую
   патологию — узаконивает ровно тот дрейф, против которого A1. Re-base и
   CUSUM — конкурирующие потребители одного сигнала.
2. **Двойной учёт с world_drift:** deadband_eff = base × f(shock_ema/shock_snap);
   плывущая база + множитель учитывают один рост шума дважды. Смесь замороженного
   якоря (shock_snap) и плавающего (база) — несогласованная семантика.
3. **Сжатие в тишине:** calm → база вниз → шум после смены режима пробивает →
   шторм. Урок уже выучен (freeze у adaptive thresholds, refractory у noise_ref).
4. **noise_ref — не прецедент:** он питает authority (гейн), а база deadband —
   порог видимости сенсора. Дрейф гейна ослабляет реакцию; дрейф порога слепит.

Если делать (только ПОСЛЕ A1): freeze при любой аномалии (calm/trip/panic/
lr_meta<1/stress), winsorize + rate-limit (паттерн noise_ref), граница
base ∈ [0.5×, 2×] warmup-снапа, τ ≥ 5–10× горизонта CUSUM.

### Б1: survival-бенчмарк (дизайн под миссию)

- **Метрика 1:** time-to-collapse (кривые выживания по сидам). Критерий
  коллапса определить явно: падение mode coverage / разгон FID / коллапс
  D-loss — зафиксировать до прогонов, не после.
- **Метрика 2 (do-no-harm + качество):** качество на согласованном горизонте
  ≥ baseline (качество не должно страдать; улучшение — бонус, не цель).
  Плюс AUC качества до коллапса («сколько полезного вытянули»).
- **Ре-анализ toy:** вывод «grav вредит на 5e-3..1e-2» получен по mode coverage
  на фикс. 40-й эпохе — метрика гонки за качеством. Карта сама отмечает: при
  80 эпохах на 5e-3 grav качественно лучше. Перепрогнать с горизонтом 80+ и
  survival-метрикой — «вред посередине» может оказаться артефактом горизонта
  (как «слепота на image» была артефактом warmup).
- Под survival-миссию ядро продукта — Panic/Climate и DS (оттягивание
  коллапса), не steering ⇒ A3 — главная работа яруса 5, не подготовка к A2.

## Дополнение 2026-06-09c: встречное ревью правок v10.9 (что они ломают)

Замер v10.8 (GB_old) vs v10.9 на идентичных потоках.

**Р-1. Warmup-фикс снял случайную защиту.** Здоровая стационарная система,
warmup=98: v10.8 degraded-инертен (LR-разброс 1e-4), v10.9 активен —
|u|=0.031, разброс LR между игроками ~2% base БЕЗ причины. Проблема A2
(«рулит без необходимости») раньше маскировалась багом warmup, теперь
операционна во всех коротко-warmup режимах. Плюс сид-лотерея калибровки:
deadband 1.25–1.92 (std 0.18/12 сидов, бывают выбросы до 3.5).
⇒ СТАТУС ГИПОТЕЗЫ (правило 2026-06-09: никаких выводов на неподтверждённых
догадках): джиттер — измеренный факт; его влияние на исход тренировки —
НЕ измерено. «Вред на toy mid-LR» — результат 40-эпохового quality-среза,
сам подлежит перемеру survival-метрикой. Порядок: СНАЧАЛА перепрогон toy +
CIFAR на v10.9 (это и есть сбор доказательств), решения по A2 — только по
их результатам. Если гипотеза вреда не подтвердится — она отбрасывается.
Независимо от гипотезы: floor на число z-сэмплов / margin ~ 1/sqrt(n)
(стыкуется с C3) — против сид-лотереи калибровки.

**Р-2. Zero-sum проекция: −17% дифференциала за точное среднее.** N=3 спайк:
спред команды 0.186→0.154, mean(lrs) 0.983→1.000. Это цена ИНВАРИАНТА
(mean=0 ∧ box ∧ полный дифференциал несовместны), не реализации. Выбор
«платить дифференциалом» спорен для дифференциального контроллера: v10.8
платил 1.7% среднего за полный дифференциал. Альтернатива: равномерный
рескейл (направление сохраняется точно, теряется магнитуда). Решить по
survival-данным; вопрос «какой инвариант правильный» — открыт.

**Р-3 (опровергнут). I-term windup через ramp от sat-фикса:** |u_i| на входе
в active и max LR-jump идентичны v10.8/v10.9 (0.006 / 10%). Не подтвердился.

**Р-4 (КРИТИЧНО, симуляцией не ловится). raise на отрицательном прокси
убивает прогон.** v10.8 молча жевал −1e-12 на шаге 50000; v10.9 роняет
300-эпоховый прогон. Прямое противоречие survival-миссии: тихая деградация
заменена смертью тренировки от FP-шума. **Поправка П1:** допуск −tol
(tol ≈ 1e-6 × масштаб + 1e-12) → clamp + одноразовый warning; raise только
на грубых нарушениях. Туда же П2: warning при коротком warmup уронит CI
с -W error — задокументировать.

**Мелкое:** reset_state теперь тихо откатывает РУЧНЫЕ мутации
dyn_deadband_z между прогонами (задокументировать); run_all сменил
сигнатуру (громко, ок); DS batch-guard закрывает в т.ч. коварный broadcast
при B=1.

**Статус поправок: П1 — обязательна до любого длинного прогона. П2 — док.
Р-1/Р-2 — решения по результатам survival-бенчмарка (Б1), Р-1 временно
парируется floor'ом на сэмплы.**

## Дополнение 2026-06-10f: критерий v2 (утверждён), probe-батарея, петля GB↔DS

### Критерий коллапса v2 — УТВЕРЖДЁН владельцем (до окончания CIFAR, потом пересмотр)

**Переносимая часть — процедура** (пре-регистрирована для будущих доменов):
1. Терминальная метрика домена → бимодальный разрыв финалов → порог смерти внутри разрыва.
2. Шум кривой качества на здоровых плато → правило подтверждения (N чекпойнтов) →
   порог просадки выше p95 подтверждённого шума.
3. Конъюнкция с независимым событием уровня сигнала (спайк лоссов).
4. TTC = первое неотыгранное событие; fallback (slow fade / край прогона):
   последняя эпоха с покрытием ≥ recovery×running-max.

**Toy-инстанс (калиброван post-hoc, числа):** смерть = финал <50 мод (разрыв 6…54);
просадка ≥30% от running max (одиночный шум p95=43%/max=85%; подтверждённый
p95=23%/max=43%; плато эквивалентности 30–40%, FP=0/24 здоровых), 2 чекпойнта,
спайк-окно −2.5/+5 эп; recovery 0.7. Валидация: 6/6 смертей, 0 ложных.
Реализован в analysis/reduce_hist.py (CRIT), вердикты в кэше/отчёте.
**CIFAR-инстанс:** терминальная метрика = D&C coverage; пороги — той же
процедурой по шуму CIFAR-прогона. После CIFAR — пересмотр всего критерия.

**Разрешение чекпойнтов:** коллапсы-с-восстановлением <5 эп невидимы на сетке 5 —
число коллапсов = метрика нестабильности (владелец), CKPT_EVERY_N_EPOCHS=2 (toy).

### Probe-батарея 10.9.3 — УТВЕРЖДЕНА

Каждые CKPT_EVERY (toy) / PROBE_EVERY=5 (image) эпох, на выделенном RNG:

## Дополнение 2026-06-13h: бэклог дальнего яруса — консенсус с санкцией, кооперативный инвариант

> Статус: идеи зафиксированы чтобы не потерялись. НЕ в работу до завершения
> отладки (план g), параметризации и абляций. Порядок ярусов не меняется.

### h1. Консенсус с санкцией (идея владельца)

Консенсус — не только мера дисбаланса, но и инструмент политики:
игрок, **систематически** тянущий систему на дно, исключается/глушится.

Эскизы механик исключения (не дизайн):

| # | Механика | Суть |
|---|----------|------|
| С1 | Вес доверия в консенсусе | w_trust_i ∈ [0,1] множителем к текущему 1/σ-весу; при w→0 игрок перестаёт искажать консенсус (не голосует), но остаётся под управлением |
| С2 | Глушение LR | lr_i → abs_lr_floor, не строгий 0 — оставить путь к воскрешению |
| С3 | Детектор систематичности | накопительный (CUSUM-подобный) счёт вклада игрока в деградацию общего тренда; НЕ мгновенный gap — транзиент ≠ вредитель |
| С4 | Амнистия | гистерезис + восстановление веса при возврате к норме. Обязательна: кратеры D-доминации восстанавливаются за ~5 эпох — перманентная казнь за транзиент была бы ошибкой |

Ограничения:
- N=2 антагонистика: убийство игрока = убийство игры (глушить D нельзя —
  исчезает обучающий сигнал G). Санкции осмысленны при N>2 и/или кооперативе.
- Зависимости: валидация zero-sum N>2 (ярус 4a/backlog), кооперативный
  инвариант (h2).

### h2. Кооперативный инвариант — зафиксированный пробел

`profile="cooperative"` сейчас меняет только integrator_mode (auto→on) и
boost_max (0.3 vs 0.1). Zero-sum проекция (`_project_zero_sum_box`)
безусловна → в «кооперативе» контроллер всё равно перераспределяет
антагонистически (буст одному = срез другому). Для честного кооператива
profile должен переключать сам инвариант: zero-sum → soft/без ограничения
среднего (common-mode рулёжка). Сейчас это заготовка режима, не режим.
Harness-сценарии cooperative/multiplayer_4 проверяют «не ломается», не
«помогает правильно».
